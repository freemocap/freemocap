# Backend Communication (Frontend Side)

The frontend talks to the Python backend over three channels: **REST** for commands, **WebSocket** for streaming data, and **Electron IPC** for desktop operations. All three are managed through the service layer in `src/services/`.

## ServerContextProvider

`ServerContextProvider.tsx` is the largest single file in the frontend (~700 lines). It manages everything that depends on the WebSocket connection. Its responsibilities grew together because they share one WebSocket, one rAF loop, and one lifecycle вҖ” but this is the current state, not the target architecture. A refactor to break it into focused pieces (connection management, frame pipeline, overlay system) is planned but deferred.

### What it owns
- `WebSocketConnection` вҖ” raw WebSocket with auto-reconnect and heartbeat
- `FrameProcessor` вҖ” binary JPEG frame parsing and decoding (in a Web Worker)
- `CanvasManager` вҖ” per-camera offscreen canvas renderers
- `OverlayManager` / `OverlayRendererFactory` вҖ” 2D overlay compositing (CharUco markers, MediaPipe skeletons)
- `FramerateStore` вҖ” observable framerate data (both backend-reported and browser-measured FPS)
- `LogStore` вҖ” buffered server logs with localStorage persistence

### What it provides (via React Context)
- `isConnected`, `connect(host, port)`, `disconnect()`
- `sendWebsocketMessage(type, payload)` вҖ” send arbitrary JSON messages
- `setCanvasForCamera(cameraId, canvas)` вҖ” register a canvas element for a camera
- `getFps()`, `getServerFps()` вҖ” read current framerate
- `subscribeToKeypointsRaw(callback)`, `subscribeToKeypointsFiltered(callback)`, `subscribeToRigidBodies(callback)` вҖ” subscribe to 3D data streams
- `setOverlayVisibility(trackerId, visible)` вҖ” toggle overlay rendering
- `trackerSchemas`, `activeTrackerId`, `getActiveSchema()` вҖ” tracker metadata

## WebSocket Connection

The WebSocket wrapper (`websocket-connection.ts`) provides:

- **Auto-reconnect** with exponential backoff: 1s вҶ’ 2s вҶ’ 4s вҶ’ 8s вҶ’ 16s (max 5 attempts, then stops)
- **Heartbeat**: ping/pong every 30 seconds to detect dead connections
- **Message dispatch**: routes binary messages to the frame processor, JSON messages through type guards

Message types are discriminated by first byte or JSON shape:

| Type | Direction | Format | Purpose |
|---|---|---|---|
| Binary frame | Server вҶ’ Client | `[msgType:1B][frameNum:4B][JPEG data...]` | Camera frames |
| Binary keypoints | Server вҶ’ Client | `[msgType:1B][keypoints:Float32Array]` | 3D keypoint positions |
| JSON: log | Server вҶ’ Client | `{type: "log", level, message, timestamp}` | Server logs |
| JSON: framerate | Server вҶ’ Client | `{type: "framerate", fps, server_fps}` | FPS metrics |
| JSON: frontend_payload | Server вҶ’ Client | `{type: "frontend_payload", payload_type, data}` | Keypoints, rigid bodies, overlays, pupil data |
| JSON: posthoc_progress | Server вҶ’ Client | `{type: "posthoc_progress", pipeline_id, phase, progress}` | Pipeline stage progress |
| JSON: tracker_schema | Server вҶ’ Client | `{type: "tracker_schema", schema}` | Tracker metadata |
| JSON: command | Client вҶ’ Server | Various shapes | Commands (connect, disconnect, etc.) |

## The rAF Processing Loop

Why `requestAnimationFrame` instead of processing WebSocket messages as they arrive?

**WebSocket messages can storm.** With N cameras at 30fps, the server sends N binary frames every ~33ms. If you process each message with `await` (for decoding), the event loop yields between messages. A storm of messages means you never finish processing one batch before the next arrives вҖ” the microtask queue fills, the main thread starves, and the UI freezes.

**rAF gives you one tick per frame.** The loop runs at display refresh rate (typically 60Hz). Each tick, it drains all queued messages, processes them, and renders. Then it waits for the next frame. This guarantees the UI stays responsive.

### Loop Structure (each tick)

```
1. ACK FRAME NUMBER
   Send the latest frame number back to the server immediately.
   This lets the server pipeline the next batch of frames while
   we decode the current batch.

2. DISPATCH JSON PAYLOADS (synchronous)
   Drain any buffered JSON messages:
   - Keypoints вҶ’ notify subscribers (Three.js update)
   - Rigid bodies вҶ’ notify subscribers
   - Overlays вҶ’ queue for overlay compositing
   - Logs вҶ’ push to LogStore
   - Pipeline progress вҶ’ dispatch to Redux
   - Framerate вҶ’ update FramerateStore

3. DECODE BINARY FRAMES (asynchronous, non-blocking)
   JPEG frames are decoded in a Web Worker so the main thread
   stays free. Decoded frames are posted back and rendered
   to their per-camera canvases.

4. MEASURE FRONTEND FRAMERATE
   Track inter-arrival times between decoded frames to compute
   the effective frontend FPS.
```

<Tip shortInfo="The frame acknowledgment (step 1) happens before decoding (step 3). This is critical: it means the backend can pipeline the next batch of frames while the frontend decodes the current batch. Without this, throughput drops by ~2x." />

## Electron IPC (tRPC)

For Electron-specific operations, the frontend uses a **tRPC proxy client** over Electron's IPC bridge.

### How it works
1. The Electron main process exposes an API via `contextBridge` as `window.electronAPI`
2. `electron-ipc-client.ts` creates a tRPC proxy that calls `window.electronAPI.invoke(path, input)`
3. Data is serialized with `superjson` (handles Dates, Maps, Sets, etc.)
4. The `isElectron()` guard lets the same code run in a browser (Electron IPC calls just no-op)

### Available operations
- File system: open folder dialogs, directory validation, read assets
- Telemetry: enable/disable
- Menu actions: forward native menu events to the renderer
- Auto-update: check for updates, download progress, install
